# Changelog

## v0.2.0 — 2026-06-28

- **Warm agent sessions.** Each agent now runs as ONE long-lived process kept warm for the whole journey — `claude` over streaming stdin/stdout, `codex` over its MCP server — instead of re-spawning the CLI every turn. No cold start and no replayed context between turns; sessions still persist to `sessions.json`, so `stop`/`start` resumes the very same conversation (re-warmed once on resume).
- **Pluggable backends.** Workers are a trait (`Backend`); adding one (gemini, kimi, …) is a single file under `src/core/worker/` plus one registry line — the dispatch and the rest never change. An unsupported `[agent]` model now fails fast in `doctor` (and the `start`/`restart` preflight) with a clear message, before any work begins.
- **Safer completion.** A run that ends with unresolved (blocked) work no longer auto-clears `.agentx/` — the reports, rounds, and state are kept for inspection; a clean run still resets as before.
- **Sharper manager review.** Each review now opens with the exact situation — which phase, which round of how many, and the named team that just handed off (e.g. `claude_1`, `codex_1`) with the paths they worked from — then judges to a tightened bar: right layer and thin-pipeline discipline (no native logic wedged high), contract/skill/design fidelity, and the run's documentation/testing policy enforced in BOTH directions (reject work the policy switched off, reject gaps it switched on). The final round's reports are archived into `rounds/` so the journey record sees everything.
- **Clearer logs.** The live status line shows the current agent · task · elapsed; the startup banner lists the engines (model · effort) actually in use. A worker stream that closes mid-turn now surfaces the real stderr.
- **`new <dir>` — bootstrap a project from an archetype.** Creates the directory, scaffolds it for a chosen archetype (`--inspire` is mandatory here — flag or menu, no auto), trains the manager, and has the manager build the runnable project skeleton from everything it trained on for the archetype. Foreground or `-b` background; `status` reports the creation phase and surfaces tool/manager errors. `init` is now a pure non-interactive scaffolder (flags only) reused by `start`/`new`; archetype + gate detection moved out of `init` into the primed run (see below). The input docs dir is a single `agentx/`, and archetypes gained a `prompts/` bucket of command guides (each a file or a folder of sub-prompts), with `overview`/`requires` templates copied into a new project's `agentx/`.
- **Audit phase + `train`/`clear` lifecycle.** A new **audit** phase runs after tasks (when `[option].audits` is on): a council of auditors examines the WHOLE built system — integration, layering, abstraction/duplication, leaked vs. adaptored providers, dangerous/abandoned dependencies, performance, and committed secrets — and raises each real defect as an explained Problem/Why/Fix remediation task; the executors build those (gate + review per task), the tasks are promoted into the main plan, and it audits again — up to `[agent].max_audits` rounds or until the system is clean. The single per-requirement decision report + training-center archiving moved out of the FSM into a reusable **`train`** command (manager writes the reports, the tool copies requires/tasks/reports into `history/`); a clean cycle auto-runs `train` then `clear`, and both are manual commands for stopping early. A gate still red after the last `max_fixes` repair now stops the run with a clear, resumable error (the only hard-stop). `[agent]` gained `max_audits` + an `audits` roster; `[option]` is `lint`/`format`/`audits`/`tests`/`fuzzes`/`benches`/`examples`/`comments`/`doc_blocks`/`doc_contracts` (the old `audits`=lint became `lint`; `audits` now toggles the phase); the gate always carries a `check` baseline plus the `lint`/`format`/`tests` pillars. The tolerance bar is factored into one shared prompt injected into every agent (ready for a future quality-level field).
- **Explicit `[option]` switches + per-option verification phases.** A new `[option]` table turns every run-shaping decision into a no-guessing flexible bool, each with its own ON/OFF prompt fragment merged into the right agents: `audits`/`formats`/`tests` (the gate's lint/format/test pillars), `comments` (inline comments on non-obvious logic, or none), and `doc_blocks`/`doc_contracts` (documentation). The old single verifier role is gone: after the tasks phase the pipeline now runs **up to four ordered verification phases — tests → benches → examples → fuzzes — each its own phase with its own `[agent]` roster, run only when its `[option]` switch is on and skipped entirely otherwise.** Each works strictly on the executed tasks, producing real project tests / benchmarks / examples / fuzzing in the language's idiomatic place (the agents pick the tooling), and is manager-reviewed every round. `[agent]` rosters are now keyed by phase (`requires`/`tasks`/`tests`/`benches`/`examples`/`fuzzes`), each accepting a bare string, a list, or empty (→ default); `manager` must resolve to exactly one agent or the config errors. `max_rounds`/`max_fixes` moved into `[agent]`; `[project]` keeps just `inspire` + a new free-text `description` (`-d`) that guides the manager when it classifies or creates a project. Every option has a global flag. Archetype `manifests/` (replacing the prompts bucket) are merged file-by-file into a new project's root on `init`/`new`, completing what's missing without overwriting.
- **Manager-led discovery, after priming.** Archetype classification and gate detection no longer run as a cold throwaway call before the run — they happen on the fully-primed manager, right after it has studied the project (the same shape as `new`: init → train the manager → confirm → act). Both prompts were stripped of language bias and hard-coded tool lists down to the essentials. The gate is built from the explicit `[option]` pillars (above) — the manager composes exactly the `audits`/`formats`/`tests` switches you set, never guessing from manifest scripts.
- **Per-requirement training memory.** The training center no longer ships a `requires` baseline that gets rebuilt every run, and no longer records one lump summary per journey. Instead, every SUCCESSFUL run archives into `~/.agentx/train/<id>/history/{requires, tasks, reports}/` — the requirements it built, the tasks they became, and ONE manager decision report per requirement (date-ordered, keyed by requirement name). Those decision reports are what future projects of the same archetype study.
- **Release hardening (final pass).** The quality gate now runs after EVERY code-writing phase — tests, benches, examples, and fuzzes as well as tasks. A producer repairs only its OWN broken artifact (test/benchmark/example/harness) and never edits project source, so a still-red gate blocks that phase for the manager instead of halting the run, while a real defect a valid test surfaces is reported. Every roster agent now carries a shared *ownership* briefing: own the whole project and treat every report — a teammate's or your own earlier round — as a claim to verify against the real code and its integration, with a manager's vigilance but never a manager's authority (persuade by evidence, not rank). The manager's per-round review verdicts are archived under `rounds/manager/` so the closing decision report can weigh them, and the priming briefing now obliges each agent to open and read every study file in full. `status` was reorganized — Workers → Sessions → Pids → a live **Now** panel — and only ever lists the backends actually on the team (one unified backend resolver). The `[option]` defaults flow from a single source, `train` archives exactly requires/tasks/reports, and the dead `.agentx/tests`/`probes` scratch dirs were removed (testers always write real project tests, never throwaway ones).
- **Internals.** `src/` reorganized so every module owns one responsibility — definitions in its `arch.rs`, behaviour split across small concern files, `mod.rs` wiring only; `core` is flat (no `support/` layer) and the worker adaptor moved to `config::worker`. Strict `core → config → app` layering, zero circular or upward dependencies. Audited production-clean: three provable `expect`s (piped stdio, runtime build), zero `unwrap`/`panic`/`todo`.

## v0.1.0 — 2026-06-24

First release. A library and binary that drives a competing team of CLI coding agents to convergence — a resumable state machine, a per-task council of agents, an upfront priming phase, and a self-training knowledge base shared across projects of the same kind.

- **Pipeline:** fixed `requires → tasks → tests → finalize`. The manager turns your requirements into an ordered backlog, architects cut it into task contracts, executors build them one at a time (a council of models, gate after every turn), verifiers attack the result — each phase manager-reviewed until it ships.
- **Priming:** the whole team studies the project once up front and confirms the bar; every later turn is a light work prompt.
- **Resumable:** the cursor (phase · task · agent · round) is checkpointed atomically after every action. `Ctrl+C`, `stop`, and `drain` halt gracefully; `start` resumes exactly where it left off.
- **Training center** (`~/.agentx/train/<archetype>/`): shared `overview · contracts · skills · designs · requires · history` per project kind, shipped in the binary. `designs/` holds frontend visual references (images, style tokens, figma exports, platform links) studied by the agents when present. The kind is auto-detected and written to `Agentx.toml`; its knowledge is injected into every briefing (yours wins on conflict), and each finished run appends one generalized lesson — so the next project of that kind starts smarter.
- **Layout:** minimal input is `Agentx.toml` + a root `Requirements.md` (an optional `agentx/`/`agents/` tree is also read). `.agentx/` is pure runtime, cleared on success; agents never write outside it. Root = nearest `Agentx.toml` upward (monorepo-safe).
- **Config** in five tables: `[project]` (`inspire`, `real_tests`, `doc_blocks`, `doc_contracts`, `max_rounds`, `max_fixes`), `[gate]` (`timeout`, `command`), `[agent]` (`timeout`, `manager`, `architects`/`executors`/`testers`), and `[claude]`/`[codex]` (`model`, `effort`). `real_tests` makes verifiers write real project tests; `doc_blocks` documents every public item; `doc_contracts` documents only the non-obvious ones — each a flexible bool, overridable per run by a flag.
- **Commands** (each a thin wrapper over the `App` library API): `init` · `start` · `restart` · `stop` · `drain` · `clear` · `ignore`/`include`/`refresh` (curate classification) · `info` · `status` (`-f/--tail` for a live dashboard) · `doctor` (preflight every required CLI) · `sync` (refresh shipped training, keep history) · `reset` · `completions` / `man` (shell completions + man page). Global flags `-i/--inspire`, `-g/--gate`, `-t/--real-tests`, `--doc-blocks`, `--doc-contracts`, `-b/--background`.
- **Resilience:** a CLI that reports an internal error is never mistaken for success. Faults are classified (transient / session / exhausted / fatal); transient faults retry with backoff, a lost session is re-trained and re-confirmed before resuming, and quota/auth failures stop cleanly and resumably.
- **Internals:** `#![forbid(unsafe_code)]` throughout; atomic state writes (tmp + fsync + rename); a liveness-checked pid lock; graceful signals via sigmask + sigwait. The embeddable type is `App`; workers live in `core::worker::Worker`.
