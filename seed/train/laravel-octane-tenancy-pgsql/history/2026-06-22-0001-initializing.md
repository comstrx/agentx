# History — foundational decisions (initializing)

This file seeds the decision log for the `laravel-octane-tenancy-api` archetype. The
training center grows its own history at runtime: after `init`, the tool appends one timestamped file
per learned decision (`YYYY-MM-DD-NNNN-slug.md`) into this same `history/` folder, shared across every
project bound to this category. This seed stays small on purpose — durable, reusable know-how belongs
in `skills/`; this folder is the running record of *why* specific calls were made.

## How to add a history entry (read before appending)
- One file per session of decisions; never edit a past file — append a new one. The filename stamp is
  the order of record.
- Inside a file, one block per decision. Keep the block schema below identical so the log stays
  scannable and diffable.
- A decision is worth a block when it constrains future work: a choice with alternatives, a tradeoff,
  or a boundary. Routine implementation is not a decision.

Block schema (copy this):

```
## DEC-NNNN — <short title>
- Decision: <the call, in one line>
- Date: <YYYY-MM-DD>
- Status: Accepted | Superseded by DEC-XXXX | Reversed
- Context: <what forced the choice>
- Rationale: <why this option, the reasoning that holds even if the title is forgotten>
- Edge cases: <what was explicitly handled / deferred and why>
- Rejected: <alternatives and the specific reason each lost>
```

## DEC-0001 — Laravel as the framework
- Decision: Build the backend on Laravel (PHP), not a bare-PHP or non-PHP stack.
- Date: 2026-06-22
- Status: Accepted
- Context: A multi-tenant, multi-vendor SaaS needs auth, queues, migrations, validation, an ORM, and a
  job/scheduler story on day one. The team's depth is in PHP. Time-to-first-revenue matters more than
  squeezing the last millisecond of latency.
- Rationale: Laravel ships the whole undifferentiated platform — Eloquent, migrations, queues, the
  scheduler, Form Requests, policies, Sanctum/Passport — as one coherent, well-documented whole. That
  removes weeks of glue and a class of integration bugs. Tenancy maps cleanly onto Eloquent global
  scopes, which is exactly the invariant this archetype is built around (see `overview/`). The
  ecosystem (Horizon, Telescope, Larastan, Pest) covers ops and quality without bespoke
  tooling (this archetype rejects the formatter, Pint — the hand-style is law; see `contracts/style.md`).
- Edge cases: Genuinely hot paths that PHP's request lifecycle handles poorly (heavy CPU work,
  long-lived connections, fan-out at extreme scale) are pushed to queued jobs or extracted into a
  separate service later — Laravel stays the system of record and the API surface, not necessarily
  every workload. This keeps the door open without paying for it now.
- Rejected:
  - **Symfony** — more assembly required for the same result; the batteries-included default is the
    point here, and the team is faster in Laravel.
  - **Node/NestJS, Go, Django** — each viable, but would trade the team's PHP fluency and Laravel's
    tenancy/queue ergonomics for a rewrite of muscle memory. No payoff that justifies the ramp.
  - **Bare PHP / micro-framework** — we'd rebuild auth, ORM, migrations, and queues by hand: more
    code to own, more places to get tenancy isolation wrong.

## DEC-0002 — Backend / API only (no bundled frontend)
- Decision: Laravel serves a versioned JSON API and nothing else — no Blade UI, no server-rendered
  pages, no bundled SPA. The frontend is a separate deployable.
- Date: 2026-06-22
- Status: Accepted
- Context: The product needs multiple clients over time (web app, mobile, vendor integrations,
  partner APIs). Coupling rendering to the backend would force one release cadence and one team
  boundary onto all of them.
- Rationale: A clean API boundary lets the frontend and backend deploy, scale, and be owned
  independently, and makes every client a first-class consumer of the same contract. It keeps Laravel
  stateless and token-authenticated, which simplifies horizontal scaling and caching. The contract
  (resource shapes, error envelope, versioning) becomes the single thing we guard — exactly what the
  `contracts/` LAW enforces.
- Edge cases: Operational/admin surfaces that legitimately want server rendering (internal dashboards,
  Horizon/Telescope) are allowed as bounded, clearly-internal exceptions — they are not part of the
  product API and never share its auth or contract. Webhooks and OAuth callbacks are API endpoints,
  not UI.
- Rejected:
  - **Monolith with Blade/Inertia** — fastest for a single web client, but bakes the web UI's release
    cadence into the backend and makes a second client (mobile, partners) a retrofit. The coupling
    cost lands exactly when the product is succeeding and hardest to undo.
  - **Bundled SPA served by Laravel** — blurs the boundary we're trying to keep sharp; deployment and
    scaling get entangled for no real gain over serving the SPA separately.
